High-level flow

OCELOT follows the same logic across species:

  1. configure the model for the chosen growth mode
  2. preprocess expression data and collapse replicates by condition
  3. build the regulatory context from expression, TF-target coefficients, and the GEM
  4. scan candidate thresholds and solve WT problems with scaled regulatory bounds
  5. choose the best threshold using a user-selected criterion
  6. branch into either:
    • knockout simulation, or
    • engineering MILP search
  7. evaluate or validate predictions if requested

Core algorithmic ideas

Reaction-level regulation from gene-level coefficients

The coefficient matrix maps TFs to target genes. OCELOT then maps target genes to reactions using the model gene-reaction structure.

Conceptually:

  1. TF expression and thresholding define a TF activity state
  2. target-level regulatory scores are computed from beta_df
  3. target genes are mapped to reactions through rxnGeneMat
  4. reaction capacities are scaled accordingly

Slack-based PROM-like relaxation

Instead of forcing every regulatory bound exactly, OCELOT introduces slack variables so that the optimization remains feasible even when the regulatory layer conflicts with metabolic constraints.

This produces a soft-constrained formulation in which:

  • kappa penalizes slack usage
  • biomass or product remains the main metabolic objective
  • the chosen WT threshold affects how aggressively the regulatory layer constrains fluxes

Engineering as controlled deviation from WT

In engineering mode, the WT TF state is used as the reference. The MILP then searches for modified TF states that:

  • satisfy minimum biomass and product requirements
  • stay close to the WT TF pattern
  • use as little slack as possible
  • respect a user-defined perturbation budget

Species-specific adaptation points

Heterotrophic organisms

Typical configuration:

  • close all exchange reactions by default
  • open a defined medium
  • open a carbon source such as glucose
  • optimize biomass

Autotrophic organisms

Typical configuration:

  • disable heterotrophic and mixotrophic biomass modes
  • open photons and carbon fixation routes
  • constrain or disable specific respiratory alternatives
  • optionally protect light, bicarbonate, and other special reactions from regulatory scaling

Why the code is split into many local functions

The current src/ layout avoids one monolithic script and instead exposes a small pipeline:

  • configuration functions define the metabolic context
  • context-building functions assemble the regulatory state
  • solver functions perform KO or engineering optimization
  • evaluation functions compute metrics and validation tables
  • utility functions keep string handling and formatting separate from modeling logic

This makes the code easier to:

  • test
  • document
  • port across organisms
  • replace piece by piece without rewriting the full workflow
