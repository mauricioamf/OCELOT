Overview

OCELOT couples a transcriptional regulatory layer to a genome-scale metabolic model so that transcription factor perturbations can be analyzed in a metabolic context.

The code supports two main tasks:

  1. TF knockout simulation
  2. Metabolic engineering strategy search

The reorganized codebase separates each logical step into a dedicated function under src/.

Core workflow

1. Prepare the metabolic model

For heterotrophic organisms, this usually means:

  • close exchanges by default
  • open a defined medium
  • open the carbon source
  • set biomass as the objective

For autotrophic organisms, this additionally means:

  • enabling photon uptake
  • constraining inorganic carbon exchange
  • disabling alternate biomass modes
  • protecting setup-specific reactions from regulatory scaling

2. Preprocess expression data

Expression tables are averaged by condition after replicate suffixes are removed from sample names.

3. Build the regulatory context

OCELOT identifies the overlap between:

  • TFs in beta_df
  • target genes in beta_df
  • genes in the expression matrix
  • genes present in the metabolic model

From there it builds:

  • filtered target expression vectors
  • filtered coefficient matrices
  • the reaction mapping matrix gene_to_rxn_map
  • the base TF expression vector for the selected WT condition

4. Scan WT thresholds

Threshold candidates are generated from either:

  • TF expression values, or
  • inferred z values

For each threshold, OCELOT builds a WT state, applies scaled reaction bounds, solves a slack-relaxed optimization problem, and records fit statistics. The best threshold is selected according to the requested criterion.

5. Run knockout or engineering mode

Knockout mode

Each TF is knocked out relative to the WT regulatory state. The resulting reaction-level capacity ratios are used to solve the KO metabolic problem.

Engineering mode

The WT TF state becomes the reference for a mixed-integer problem that searches for perturbation strategies under growth and product requirements.

6. Evaluate or validate predictions

KO results can be evaluated against a TF essentiality table. Engineering strategies can be validated by fixing the engineered TF state and re-optimizing growth and product, with optional single-KO validation for TFs that appear in the strategy set.

Main outputs

Knockout outputs

Common knockout outputs include:

  • WT_results
  • rmse_table
  • best_threshold
  • KO_growth
  • KO_fluxes
  • has_coverage
  • covered_TF_list
  • metrics_all
  • metrics_cutoff

Engineering outputs

Common engineering outputs include:

  • WT_results
  • resultsTable
  • filteredResultsTable
  • final_results
  • sKO_results

Parameter notes

Threshold behavior

  • thresholdSource = 'tf_expression' is common for heterotrophic organisms
  • thresholdSource = 'z_vector' is often useful in autotrophic setups

Threshold selection behavior

Common options include:

  • RMSE
  • Growth
  • Discriminability
  • Composite

Evaluation behavior

  • performEssentialityEvaluation = true enables KO benchmarking
  • validateStrategies = true enables engineering re-solving for predicted multi-TF strategies
  • validateSingleKOs = true enables extracted single-KO checks

Practical guidance

When to use the unified entry point

Use OCELOT when you want one clean top-level call and centralized mode selection.

When to call lower-level functions directly

Call lower-level functions when you want to:

  • debug a single stage
  • benchmark alternative threshold strategies
  • replace one solver step without changing the rest of the workflow
  • build custom wrappers for a new organism
