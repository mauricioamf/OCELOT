Overview

This block of the API prepares the wild-type model, builds the regulatory mapping from expression and beta coefficients, scans threshold candidates, and selects the best threshold for downstream knockout or engineering analysis.

In practice, this stage defines the WT context reused by the rest of OCELOT.

configureBaseModel

Build the species and growth-mode-specific baseline model for standard workflows. This typically includes opening the chosen medium, setting objective reactions, applying glucose uptake when relevant, and protecting reactions that should not be rescaled by regulatory logic.

Typical inputs.

  • COBRA model
  • params.growthType
  • params.biomassRxn
  • params.mediumRxns
  • params.glucoseRxn and params.glucoseUptake
  • optional bound-change list

Typical outputs. configured WT model, biomass index, WT FBA solution, and bookkeeping fields such as protected reaction indices.

configureAutotrophicModel

Apply autotrophic-specific preparation that would be awkward to pack into a generic base-model helper. This is especially relevant for cyanobacterial workflows where photons, bicarbonate, CO2, and photosynthetic or respiratory reaction edits must be coordinated carefully.

Typical behavior.

  • activate the autotrophic biomass objective
  • disable mixotrophic or heterotrophic biomass reactions
  • open photon uptake
  • set bicarbonate and CO2 exchange assumptions
  • disable reactions that should be inactive under the chosen setup
  • optionally perform auxiliary optimizations to estimate photon demand

preprocessExpressionData

Prepare condition-level expression tables from raw replicate-level input. This function usually strips replicate suffixes, groups samples by condition name, and collapses replicates using the package’s chosen summary rule.

Why it matters. Threshold scanning and downstream condition matching rely on consistent condition labels.

buildRegulatoryContext

Construct the regulatory objects needed by the rest of the pipeline. This generally includes:

  • TF list
  • target gene list
  • filtered beta matrix aligned to model genes
  • target expression vector for the selected reference
  • reaction-to-target mapping via rxnGeneMat

Typical outputs. A single context struct that lets the rest of the pipeline avoid repeated remapping work.

chooseThresholdValues

Generate threshold candidates. Depending on params.thresholdSource, thresholds may come from TF expression values or from reconstructed latent TF activity values such as z_vector.

Common choices.

  • 'tf_expression' for simpler expression-threshold workflows
  • 'z_vector' for workflows that threshold the inferred TF activity state

buildThresholdState

Given one threshold candidate, build the threshold-specific regulatory state. This usually includes:

  • binarized TF state
  • continuous or masked TF activity vector
  • target-level regulatory effect
  • reaction-level activity vector
  • normalized WT scaling table

This function is the bridge between abstract threshold candidates and the concrete regulatory scaling used by optimization.

runWTThresholdSelection

Scan thresholds and solve the WT scaled-bound problems needed to evaluate them. The main purpose is to determine which threshold yields a reasonable compromise among fit quality, growth retention, and optional discriminability criteria.

Typical tasks.

  • loop over threshold candidates
  • build threshold-specific WT scaling
  • solve slack-penalized optimization problems
  • store biomass, kappa, and target reconstruction information
  • track infeasible runs and stop early if needed

selectBestThreshold

Choose the final threshold from the threshold scan. Supported logic commonly includes:

  • 'Growth' for highest feasible WT growth
  • 'RMSE' for best target reconstruction
  • composite criteria when available

Typical output. best threshold identifier plus summary fields such as RMSE, R², growth, and selected kappa.
